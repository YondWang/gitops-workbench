# 代码合入与版本管理规则说明

## 1. 文档目的

本文用于说明当前版本管理工具中的代码分支、代码合入、版本号生成和发版规则，便于团队成员统一理解和执行。

当前推荐的管理方式是：**以 Web 页面作为主要操作入口，通过 GitLab REST API 完成分支、Tag 和合入管理**。仓库中历史 Pipeline 代码保留，但不再作为日常版本管理入口。

## 2. 总体原则

1. 所有版本相关操作尽量通过 Web 页面完成，减少手工创建分支、手工填写版本号和手工打 Tag。
2. 分支命名必须统一，便于工具识别、筛选和自动生成后续分支。
3. 版本号采用四段式：`A.B.C.D`，例如 `3.2.0.0`。
4. Web 工具负责自动生成 Baseline 版本号、Fix RC 分支序号和默认 Tag 名称。
5. 发版时由 Fix 分支打 Tag，并同步合入对应 Baseline。
6. 如果合入存在冲突，不自动强行处理，由负责人在 GitLab MR 中手动解决。
7. 当前 GitLab Pipeline 仅保留代码，不自动触发，不作为版本管理执行入口。

## 3. 角色与权限

当前 Web 工具分为两个角色：

| 角色 | 主要权限 | 适用人员 |
|---|---|---|
| `user` | 查看分支、查看 Tag、创建 Feature 分支 | 普通开发人员 |
| `admin` | `user` 的全部权限，加上初始化 Baseline、创建 Fix、打 Tag、发版同步、仓库管理 | 版本管理员、发布负责人 |
| `待补充` | 当前admin暴露太多权限，可能导致操作混乱 | 

建议：

- 普通开发只使用 `user` 角色创建 Feature 分支。
- Baseline、Fix、Tag、发版合入由 `admin` 统一操作。
- 多仓库批量操作前，必须确认所有目标仓库处于同一版本管理节奏。

## 4. 分支命名规则

### 4.1 Baseline 分支

Baseline 表示某个版本的稳定基线。

格式：

```text
baseline/{version}
```

示例：

```text
baseline/3.2.0.0
baseline/3.3.0.0
```

用途：

- 作为 Feature 分支的来源。
- 作为 Fix 分支的来源。
- 作为发版后 Fix 内容同步回来的目标分支。

### 4.2 Feature 分支

Feature 表示新功能开发分支。

格式：

```text
feature/{version}/{TASKID-desc}
```

示例：

```text
feature/3.2.0.0/TASK1234-user-login
feature/3.3.0.0/TASK5678-map-update
```

规则：

- Feature 必须从对应 Baseline 拉出。
- `TASKID` 使用任务号，便于追踪需求来源。
- `desc` 使用简短英文或拼音描述，避免中文、空格和特殊符号。
- Feature 完成后，由开发流程合入目标分支，Web 工具当前主要负责创建分支。

### 4.3 Fix 分支

Fix 表示候选发布或修复验证分支。

格式：

```text
fix_{version}/rcN
```

示例：

```text
fix_3.2.0.0/rc1
fix_3.2.0.0/rc2
```

说明：

- `rc` 是 Release Candidate 的缩写，表示“候选发布版本”。
- `rc1` 表示第 1 个候选发布版本。
- 如果验证不通过，需要继续修复，则创建 `rc2`、`rc3`，以此类推。
- 工具会根据当前已有 Fix 分支自动计算下一个 RC 序号。

为什么使用 `fix_3.2.0.0/rcN` 而不是 `fix/3.2.0.0-rcN`：

- Git 中如果已经存在名为 `fix` 的分支，就不能再创建 `fix/...` 这种层级分支。
- 使用 `fix_{version}/rcN` 可以避免与已有 `fix` 分支冲突。

## 5. 版本号规则

### 5.1 四段式版本

版本号格式：

```text
A.B.C.D
```

示例：

```text
3.2.0.0
3.2.1.0
3.3.0.0
```

字段含义：

| 字段 | 名称 | 触发场景 |
|---|---|---|
| `A` | 主版本 | 架构不兼容、重大系统调整 |
| `B` | 次版本 | 向下兼容的新功能基线 |
| `C` | 补丁版本 | Bug 修复、小优化、补丁发布 |
| `D` | 构建/热修复号 | 同一补丁下的构建、临时热修复 |

### 5.2 自动生成规则

初始化 Baseline 时，版本号可以不手动填写。

工具会扫描当前仓库已有分支和 Tag 中的最大四段式版本，然后根据“变更类型”自动生成下一个版本号。

| 变更类型 | 规则 | 示例：当前最大版本 `3.2.0.0` |
|---|---|---|
| `major` 架构不兼容 | `A+1`，`B/C/D` 清零 | `4.0.0.0` |
| `minor` 新功能基线 | `B+1`，`C/D` 清零 | `3.3.0.0` |
| `patch` 修复/优化 | `C+1`，`D` 清零 | `3.2.1.0` |
| `build` 构建/热修复 | `D+1` | `3.2.0.1` |

推荐使用方式：

- 新一轮功能开发：选择 `minor`。
- 已有版本上的缺陷修复：选择 `patch`。
- 同一补丁版本下重复构建或小范围热修复：选择 `build`。
- 架构、协议、平台存在不兼容变化：选择 `major`。

## 6. 标准操作流程

### 6.1 初始化 Baseline

适用场景：

- 新版本周期开始。
- 需要从 `main`、已有稳定分支或 Tag 创建一个新的版本基线。

操作流程：

1. 使用 `admin` 登录 Web 工具。
2. 进入 `Baseline` 页面。
3. 选择来源分支或 Tag。
4. 选择变更类型，例如 `minor`、`patch`。
5. 版本号可以留空，由系统自动生成。
6. 点击创建，生成：

```text
baseline/{version}
```

示例：

```text
baseline/3.3.0.0
```

### 6.2 创建 Feature 分支

适用场景：

- 开发新功能。
- 开发任务需要从指定 Baseline 开始。

操作流程：

1. 使用 `user` 或 `admin` 登录 Web 工具。
2. 进入 `Feature` 页面。
3. 选择目标 Baseline。
4. 填写任务号和简短描述。
5. 点击创建，生成：

```text
feature/{version}/{TASKID-desc}
```

示例：

```text
feature/3.3.0.0/TASK1234-user-login
```

### 6.3 创建 Fix 分支

适用场景：

- 某个 Baseline 准备进入发版验证。
- 需要从 Baseline 拉出候选发布分支。

操作流程：

1. 使用 `admin` 登录 Web 工具。
2. 进入 `Fix` 页面。
3. 选择目标 Baseline。
4. 点击创建。
5. 系统自动计算下一个 RC 序号，生成：

```text
fix_{version}/rcN
```

示例：

```text
fix_3.3.0.0/rc1
```

如果已经存在 `fix_3.3.0.0/rc1`，再次创建会生成：

```text
fix_3.3.0.0/rc2
```

### 6.4 Fix 打 Tag 并同步 Baseline

适用场景：

- Fix 分支验证通过，准备发版。
- 需要保留发布 Tag，并将 Fix 内容同步回 Baseline。

操作流程：

1. 使用 `admin` 登录 Web 工具。
2. 进入 `发版` 页面。
3. 选择 Fix 分支。
4. 选择目标 Baseline。
5. Tag 名称可留空，系统会生成默认 Tag。
6. 填写发版说明。
7. 点击创建 Tag 并同步 Baseline。

默认 Tag 示例：

```text
v3.3.0.0-rc1
```

系统动作：

1. 在 Fix 分支上创建 Tag。
2. 创建 `Fix -> Baseline` 的 Merge Request。
3. 尝试自动合入。
4. 如果无冲突，完成同步。
5. 如果有冲突，保留 MR，由负责人手动解决。

## 7. 代码合入规则

### 7.1 Feature 合入

Feature 分支用于新功能开发。

建议规则：

- Feature 必须从对应 Baseline 创建。
- 开发完成后，通过 MR 合入团队约定的目标分支。
- 合入前需要完成代码评审和基本验证。
- 不建议直接把未验证 Feature 合入 Baseline。

### 7.2 Fix 合入

Fix 分支用于候选发版和修复验证。

规则：

- Fix 从 Baseline 创建。
- Fix 验证通过后才能打 Tag。
- 发版时必须同步回原 Baseline，保证 Baseline 包含已发布内容。
- 如果自动合入失败，必须在 GitLab MR 中手动解决冲突。

### 7.3 Baseline 保护

Baseline 是版本基线，不建议随意直接提交。

建议规则：

- 禁止开发人员直接 push 到 Baseline。
- Baseline 更新应通过 MR 或 Web 工具发版同步完成。
- 每次 Baseline 更新后，要能对应到 Fix、Tag 或明确的 MR。

## 8. 多仓库管理规则

当前工具支持管理多个 GitLab 仓库。

操作范围有两种：

| 操作范围 | 含义 |
|---|---|
| 当前仓库 | 只对页面当前选中的仓库执行 |
| 全部启用仓库 | 对仓库管理中所有启用仓库执行 |

批量操作策略：

1. 先对所有目标仓库做预检查。
2. 只要有任一仓库预检查失败，就不执行实际操作。
3. 所有仓库预检查通过后，再统一执行。
4. 页面日志会显示每个仓库的执行结果。

适用建议：

- 日常单仓库操作使用“当前仓库”。
- 多仓库统一建 Baseline 或统一拉 Fix 时，才使用“全部启用仓库”。
- 批量操作前先确认每个仓库的来源分支、Baseline 和版本节奏一致。

## 9. Pipeline 使用策略

当前仓库保留历史 Pipeline 代码，但不再自动触发。

`.gitlab-ci.yml` 中已加入：

```yaml
workflow:
  rules:
    - when: never
```

效果：

- push 不触发 Pipeline。
- merge 不触发 Pipeline。
- tag 不触发 Pipeline。
- schedule 不触发 Pipeline。
- Pipeline 文件仍保留在仓库中，后续需要恢复时可以调整该规则。

当前推荐：

- 日常版本管理使用 Web 页面。
- Pipeline 代码作为历史方案和后续扩展储备保留。
- 不再通过 GitLab Pipeline 执行分支创建、Tag 创建和版本管理操作。

## 10. 常见问题

### 10.1 为什么版本号可以不填？

工具会从已有分支和 Tag 中提取最大四段式版本，然后根据变更类型自动生成下一个版本号。

这样可以避免人工手填错误，例如把 `3.2.1.0` 写成 `3.2.0.1`。

### 10.2 RC 是什么意思？

RC 是 Release Candidate，表示候选发布版本。

例如：

```text
fix_3.3.0.0/rc1
```

表示 `3.3.0.0` 版本的第一个候选发布分支。

### 10.3 为什么 Fix 分支不用 `fix/...`？

因为 Git 不允许同时存在 `fix` 分支和 `fix/...` 层级分支。

为了避免与已有 `fix` 分支冲突，统一使用：

```text
fix_{version}/rcN
```

### 10.4 Tag 名称为什么可以留空？

发版时如果不填写 Tag 名称，工具会根据 Fix 分支自动生成。

例如：

```text
fix_3.3.0.0/rc1
```

默认生成：

```text
v3.3.0.0-rc1
```

### 10.5 自动合入失败怎么办？

自动合入失败通常是代码冲突。

处理方式：

1. 打开页面日志中的 MR。
2. 在 GitLab 中查看冲突文件。
3. 由负责人手动解决冲突。
4. 冲突解决后继续合入。

## 11. 重点总结

团队需要统一记住以下几点：

1. Web 页面是当前版本管理的主要入口。
2. Baseline 是版本基线，Feature 和 Fix 都从 Baseline 派生。
3. 版本号是四段式 `A.B.C.D`，优先由系统自动生成。
4. Fix 分支统一使用 `fix_{version}/rcN`。
5. 发版动作是“Fix 打 Tag，并同步回 Baseline”。
6. 有冲突时不自动强行合入，必须通过 GitLab MR 手动解决。
7. Pipeline 文件保留，但不再自动触发。
