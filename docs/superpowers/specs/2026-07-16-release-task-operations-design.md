# 发版任务页面：运行记录清理与界面优化设计

## 背景与目标

发版任务页面的“最近运行”列表目前只能查看，不能删除历史记录；页面桌面端使用非对称双栏，左侧表单又受最大宽度限制，造成右侧内容区偏窄且整体没有充分利用页面空间。构建配置复选框继承了通用输入框尺寸，显示为过大的整行控件。

本次改造的目标是补齐运行历史管理能力，并在不改变既有发版、Tag、云盘产物与权限模型的前提下，改善该页面的布局和控件观感。

## 范围

### 包含

- 管理员可删除单条发版运行记录。
- 管理员可在确认后清空全部发版运行记录。
- 桌面端发版任务页面改为等宽双栏，左右两个主面板撑满工作区；窄屏保持单列。
- 构建配置改为紧凑、可点击、带选中态的复选项。
- 为新增删除逻辑编写后端自动化测试，并对页面进行本地验证。

### 不包含

- 删除 GitLab Tag、Pipeline、Merge Request、云盘目录或构建产物。
- 修改发版计划、任务保存、继续发版和构建触发的业务逻辑。
- 为历史运行记录引入自动过期策略或恢复站。

## 方案选择

采用“平衡双栏 + 就地可控清理”方案。

页面继续保留现有的左侧“自动任务管理”与右侧“任务列表、手动执行、最近运行”分组，但在桌面端将两列改为等宽，并取消发版任务区域左侧表单卡片的 720px 最大宽度限制。这样既保留现有用户操作路径，也能完整利用工作区。

“最近运行”在现有列表内增加操作列，并在标题栏加入“清空全部”按钮。单条删除和清空全部均使用浏览器确认提示，防止误操作。

## 接口与数据流

运行记录继续存储在 `webapp/data/release_runs.json`，数据结构不变。

新增两个受 `admin` 权限保护的接口：

- `DELETE /api/release-runs/{run_id}`：删除指定运行记录。
- `DELETE /api/release-runs`：清空全部运行记录。

后端在读取现有记录、筛选后调用既有 `save_release_runs` 持久化，并返回最新的 `runs` 列表。找不到指定记录时返回清晰的参数错误。删除操作仅影响本地历史 JSON；已触发的 GitLab 操作与产物不会被修改。

前端点击删除按钮后：确认 -> 调用对应接口 -> 写入操作日志 -> 用响应中的 `runs` 刷新 `state.scheduleRuns` -> 重新渲染列表。清空操作同样刷新列表。事件委托仍绑定在“最近运行”表格主体，标题按钮单独绑定。

## 页面设计

### 布局

- `#schedules` 内的双栏网格在宽屏使用两个 `minmax(0, 1fr)` 等宽轨道；两个面板均可拉伸至列宽。
- 视口宽度低于现有响应式断点时，双栏退化为一列，保持当前移动端阅读体验。
- 运行记录表增加“操作”列；状态、Tag、来源和云盘目录的现有信息保留。
- 标题栏将“刷新”与“清空全部”组合为紧凑操作区，危险操作使用现有红色危险按钮语义。

### 构建配置

- 为 `inline-options` 定义独立样式，不继承通用 `input` 的全宽和 40px 最小高度。
- 每一项使用横向 `inline-flex` 布局：固定 18px 复选框、标签文字与适当内边距。
- 选项容器使用浅色背景、圆角与边框；勾选项表现为青绿色强调态。键盘聚焦仍清晰可见。
- 两处构建配置（自动任务管理、手动执行）复用相同 HTML 结构与样式。

## 错误处理与权限

- 删除接口统一沿用现有 Handler 的 `admin` 鉴权；没有管理员权限的请求将得到现有的认证/授权响应。
- 未找到运行 ID 时不写入文件，并返回明确错误信息。
- 清空空列表是幂等操作，成功后返回空数组。
- 前端请求失败时沿用 `appendLog` 输出错误，保留当前渲染内容。

## 测试与验收

后端测试覆盖：

1. 删除一条记录只移除目标记录，其他记录保持不变。
2. 删除不存在的记录返回错误。
3. 清空全部记录后持久化结果为空。
4. 现有删除任务逻辑仍只清理与该任务关联的运行记录。

页面验收覆盖：

1. 管理员可在最近运行中删除单条记录，列表立即更新。
2. 管理员可清空全部记录，列表显示空状态。
3. 宽屏中左右主面板等宽且填满工作区；窄屏自动单列。
4. 两处构建配置中的复选框为紧凑尺寸，选中态与键盘焦点可辨识。
